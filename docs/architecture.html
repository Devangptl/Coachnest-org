Coachnest is a full-stack online learning platform built on Next.js 15 App Router. It supports three distinct user roles (Student, Instructor, Admin), each with their own portal. The platform handles course creation and delivery, Stripe payments with dynamic revenue splits, a gamification engine, community tools, and blog/marketing content.

рџ”Ђ Layer Architecture

The codebase is split into five clear layers with a strict separation of responsibilities.

рџ›ЎпёЏ Layer 1 вЂ” Edge Middleware (middleware.ts)
Runs on every request before rendering. Refreshes Supabase JWT, reads role from app_metadata, blocks wrong-role routes, redirects unauthenticated users.
рџ–ҐпёЏ Layer 2 вЂ” Server Components (RSC)
Page-level components that fetch data directly from Prisma on the server. No API round-trip. Data is rendered as HTML and streamed to the browser.
рџЊђ Layer 3 вЂ” Route Handlers (app/api/)
Thin HTTP handlers. Parse request в†’ validate with Zod в†’ call getSession() в†’ delegate to service в†’ return NextResponse.json(). No business logic lives here.
вљЎ Layer 4 вЂ” Services (services/)
All business logic. Called from route handlers and server components. Examples: payment.service.ts (revenue splits), refund.service.ts (eligibility + Stripe refunds), quiz.service.ts (scoring).
рџ—„пёЏ Layer 5 вЂ” Lib + Prisma (lib/)
Singleton clients and pure utilities: lib/prisma.ts (Prisma singleton), lib/auth.ts (getSession()), lib/gamification.ts (awardXp, grantBadge), lib/email.ts (Resend templates), lib/stripe.ts.

рџ”ђ Authentication

Authentication is handled by Supabase Auth using HttpOnly session cookies. User metadata is split between two Supabase fields:

Supabase FieldContents
auth.users.app_metadata{ role: "STUDENT" | "INSTRUCTOR" | "ADMIN" }
auth.users.user_metadata{ name, avatar }

Session Flow

1
User signs up or logs in
POST /api/auth/signup or /api/auth/login вЂ” Supabase creates the user and sets app_metadata.role.
2
HttpOnly cookie set
Supabase issues a JWT stored in an HttpOnly session cookie вЂ” never accessible to JavaScript.
3
Edge middleware refreshes token
On every request, middleware calls the Supabase server client to silently refresh the JWT if it's close to expiry.
4
Route handlers call getSession()
lib/auth.ts reads the current session and returns a SessionPayload: { userId, email, role, name, avatar }.
5
Role checked before mutation
Route handlers re-check the role even after middleware, as the API can be called directly.

Password Reset Flow

POST /api/auth/forgot-password  в†’  Supabase sends reset email
User clicks link в†’ /reset-password page
POST /api/auth/reset-password   в†’  Supabase updates password

рџ’і Payment & Revenue System

Checkout Flow

1
Student initiates purchase
POST /api/payments/create-payment-intent вЂ” validates coupon / referral code, calculates final price, creates a Stripe Checkout Session and a pending Order row.
2
Stripe Checkout
User completes payment on Stripe's hosted page. On success, redirected to /checkout/success.
3
Payment verified
POST /api/payments/verify вЂ” confirms the session was paid, creates the Enrollment, credits InstructorWallet, writes LedgerEntry rows, sends the purchase email, and awards gamification XP.
4
Stripe webhook (async)
POST /api/payments/webhook вЂ” handles late events and edge cases. Signature is verified against STRIPE_WEBHOOK_SECRET before processing.
